1. Выносить общие проверки в метод (через внешнюю переменную)!
	Это могут быть проверки по наличию полей созданной сущности (допустим 5 пользователей созданые в разных местах, будут требовать один и тот-же набор проверок). 
		Такие проверки нужно выносить в отдельные методы.		
				???Еще ведь вопрос, как делать проверки в том же StopLight если нужно выполнить одну и туже проверку в рамках разных запросов.
2. Выносить все подготовительные действия в отдельные методы (через внешнюю переменну)!
	Это характерно для Postman - так как переиспользование ранее написанных запросов - не возможно. Приходится для подготовки данных использовать - обычные методы.
3. Всем переменным давать узнаваемые и уникальные наименования (наименования, по которым можно будет определить от куда эта переменная) да проще менять - можн в sublime через find-replace.
4. Нарезать на мелкие части коллекции!
	- Проще проводить Review при commite.
	- Проще править и поддерживать.
		Не всегда получится переиспользовать те-же коллекции для повторного использования - так-как где-то могут быть тесты в части проверки кол. объектов итд. Хотя можно добавлять проверку некого маркера - и если есть маркер, то такие тесты не проводить. А проводить тесты только на базе id.
	
5. Для сущностей - если это иерарихя - лучше использовать цифровые префиксы, а в качестве текста уже можно использовать вменяемые пояснения.
	К примеру: "1_1_Подзадача_созданная_после_запуска_проекат".
	5.1. Для элементов, что будут отображаться на верхнем уровне, для понимания где это было создано можно использовать pm.info.requestName.	
	5.2. В случае множественно лучше использовать миллисекунды + допустим инкремент. Разумеется это подходит для 5.1. но не подходит для 5.
